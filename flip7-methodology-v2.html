Flip 7 方法论总结 V2

基于完整对话过程的工程化实践抽象，从"用户想做个游戏"到"可交付的项目"。

1. 对话过程回顾

阶段一：需求澄清
用户提出模糊需求 → 逐步澄清规则 → 确认核心玩法
阶段二：原型验证
生成可运行原型 → 用户测试 → 发现规则理解偏差 → 修正逻辑
阶段三：文档固化
整理PRD → 固化验收标准 → 建立自检清单
阶段五：视觉迭代
卡牌设计系统 → UI风格统一 → 交互细节优化
阶段六：正式开发
迁移到微信小游戏 → 真机测试 → 提交审核

2. 核心方法论

2.1 对话式需求工程

传统方式 vs 对话式

  • 传统：一次性写完PRD → 开发 → 发现偏差 → 返工
  • 对话式：逐步澄清 → 原型验证 → 即时修正 → 减少返工
价值：需求变更成本从"开发后"降到"开发前"，节省70%返工时间。

2.2 渐进式细化

模糊想法

  • "想做个Flip 7小游戏"
  • 只有一句话需求

具体规则

  • 牌堆构成、判负条件、计分规则
  • 边界情况处理

可执行代码

  • 纯函数模块、状态机、UI渲染
  • 可测试、可验证

2.3 可视化验证

  • HTML原型让规则可以被直接体验，而不是被"理解"
  • 用户测试原型时，能发现规则描述的歧义
  • 原型是活的文档，比静态PRD更有效
实践：先写"可运行的规则书"（原型），再写"可阅读的规则书"（PRD）。

2.4 反馈驱动迭代

用户反馈 → 分析问题 → 修复/优化 → 重新验证 → 用户确认
  • 用户的每次反馈都是改进机会，不是"找茬"
  • 快速响应反馈，让用户有参与感
  • 每次修改都要验证，确保没有引入新问题

2.5 文档即过程

过程产物

  • PRD：记录规则决策
  • 架构图：记录技术决策
  • 自检清单：记录验收标准

价值

  • 可追溯：每个决策都有依据
  • 可交接：新人能快速理解
  • 可复用：下次类似项目能参考

2.6 版本边界管理

  • 明确V0/V1/V2各自做什么、不做什么
  • 新想法一律记入backlog，不进当前版本
  • 用Git管理版本演进，每个版本有明确的commit
原则：版本边界是"产品边界"的体现，不是"时间边界"。

2.7 逻辑与视图分离

flip7-v05.html（逻辑 + 视图内联）→ 单文件自包含原型
  • 核心规则因模块未成功集成，目前与 UI 同文件维护
  • 适用于单页原型/小项目，便于快速迭代和分享
  • 后续若迁移微信小游戏，再抽离成独立逻辑层

2.8 边界条件显式化

  • 把"abnormal path"列成checklist
  • 每个边界在原型里走一遍
  • 用自检清单固化，每次发版前必跑
经验：边界条件是bug的高发区，必须显式处理。

4. 可复用的检查清单

启动项目前

  • □ 核心规则是否已量化（布尔条件+后果）
  • □ 边界条件是否已列出
  • □ 版本边界是否已定义
  • □ 是否有可运行的原型

每次修改后

  • □ 是否验证了原有功能
  • □ 是否引入了新问题
  • □ 是否更新了相关文档
  • □ 是否提交了Git commit

发版前

  • □ 自检清单是否全部通过
  • □ 边界条件是否已测试
  • □ PRD是否已更新
  • □ 版本号是否已更新

5. 一句话总结

从对话到产品，核心是：
1. 用对话澄清需求，而不是一次性定义
2. 用原型验证规则，而不是用文档想象
3. 用文档固化决策，而不是依赖记忆
4. 用Git管理演进，而不是手动复制
5. 用测试保障质量，而不是靠人工检查
